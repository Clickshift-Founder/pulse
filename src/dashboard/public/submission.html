✅ Create a wallet programmatically

Pulse creates all six agent wallets programmatically using Keypair.generate() from @solana/web3.js — the same cryptographic standard used by Phantom, Backpack, and every institutional custody solution on Solana. This happens automatically at server startup, before any user opens the dashboard. The AgentWallet.loadOrCreate() method checks for an existing encrypted wallet file; if none exists, it generates a fresh Ed25519 keypair, encrypts the private key with AES-256-GCM, and writes the ciphertext to disk.

// AgentWallet.ts — programmatic wallet creation with HKDF key derivation const keypair = Keypair.generate(); // Ed25519 const perAgentSalt = crypto.randomBytes(32).toString("hex"); // unique per wallet const agentKey = crypto.hkdfSync("sha256", MASTER_KEY, salt, info, 32); const { encrypted, iv, authTag } = aesGcmEncrypt(privateKey, agentKey); agentKey.fill(0); // zero key from memory immediately fs.writeFileSync(walletPath, JSON.stringify({ agentId, publicKey, encryptedPrivateKey, iv, authTag, keyVersion: 2, perAgentSalt // salt on disk; MASTER_KEY never on disk }));

✅ Sign transactions automatically

Every transaction in Pulse is signed by the agent's own keypair — autonomously, with zero human approval. The private key is decrypted in-memory only for the duration of signing, then released. It never appears in logs, never leaves the machine, and is never transmitted to any third party.

// In-memory signing — private key exists for milliseconds only const keypair = this.decryptKeypair(); // AES-256-GCM decrypt transaction.sign(keypair); // Ed25519 sign // keypair goes out of scope — GC'd immediately const sig = await connection.sendRawTransaction(transaction.serialize());

Part 2

Technical Deep Dive

Wallet Design & Cryptography

The AgentWallet class is the foundational primitive of the entire system. Every agent is, at its core, a wallet with a reasoning layer attached to it. The design philosophy is isolation-first: no agent can access another agent's key material under any circumstances. Pulse implements a security model that goes beyond what most agentic wallet prototypes demonstrate.

Keypair Generation. Wallets use Ed25519 elliptic curve cryptography via Keypair.generate() from @solana/web3.js — the same cryptographic foundation used by hardware wallets and institutional custody solutions. The public key is derived deterministically and stored plaintext. The private key never exists in plaintext on disk at any point.

Two-Layer HKDF Key Derivation (v2). This is the security feature that distinguishes Pulse architecturally. Many implementations encrypt all agent wallets with the same derived key — meaning cracking one cracks all. Pulse instead uses HKDF-SHA256 (RFC 5869) to derive a unique 32-byte encryption key per agent, with three independent inputs:

  • MASTER_KEY — hardened from ENCRYPTION_SECRET env var using scrypt (N=32768, r=8, p=1). Only in environment, never on disk.
  • agentId — the agent's identifier, bound into the HKDF info field as "pulse:{agentId}:v2"
  • perAgentSalt — 32 bytes of cryptographically random entropy generated at wallet creation, stored in the wallet file

An attacker who steals a wallet file gets the perAgentSalt — but that alone is useless. They also need the MASTER_KEY (which only exists as an environment variable on the server) and the specific agentId. Crucially, there is no AEK file stored on disk — the per-agent key is derived on demand, used for milliseconds, and then explicitly zeroed from memory with .fill(0). Nothing is stored. Nothing persists.

// HKDF derives a unique key for each agent — never stored, always computed const info = Buffer.from(`pulse:${agentId}:v2`); const agentKey = crypto.hkdfSync("sha256", MASTER_KEY, perAgentSalt, info, 32); const cipher = crypto.createCipheriv("aes-256-gcm", agentKey, iv); // ... encrypt ... agentKey.fill(0); // zero key material from memory immediately

MASTER_KEY Hardening. The raw ENCRYPTION_SECRET env string is not used directly. It is first hardened through scryptSync with OWASP-recommended parameters (N=32768, r=8, p=1), making brute-force dictionary attacks computationally prohibitive even if an attacker knows the KDF scheme.

Agent Isolation. Each of the six agents has a completely independent encrypted wallet file and a unique derived encryption key. Compromise of one wallet file exposes nothing about any other agent — not through the file itself, not through the salt, not through any stored material. The blast radius of a key compromise is exactly one agent.

Key Rotation. Pulse includes a live key rotation endpoint (POST /api/security/rotate-keys). After rotating the ENCRYPTION_SECRET in the environment, a single API call re-derives all agent keys with the new MASTER_KEY and fresh per-agent salts — no agent downtime, no manual re-encryption, no key files to handle. Old wallets on disk with legacy encryption (v1, single shared key) are automatically detected and upgraded to v2 HKDF on the next load.

The Governor: 7-Layer Safety System

The Governor is the most architecturally important component. It has no wallet, no agency, and no opinions about strategy. It is a stateless gatekeeper that sits between every AI decision and every wallet signature. The design principle: the AI cannot harm the user even if it hallucinates.

The Heartbeat Architecture — What Makes Agents Alive

A bot reacts to commands. An agent wakes up on its own, perceives its environment, and acts. Every Pulse agent runs an independent HeartbeatEngine — a self-managing interval loop that fires autonomously every 45–120 seconds depending on the agent's role. Each heartbeat cycle follows a fixed six-step protocol:

  1. WAKE — Agent comes online, checks running state, logs cycle number
  2. READ — Reads HEARTBEAT.md directive file and fetches live market prices
  3. THINK — Routes to Clickshift proprietary intelligence with full portfolio context for AI reasoning
  4. PLAN — Decides on actions based on AI output (rule-based fallback if no API key)
  5. EXECUTE — Submits planned actions to the Governor for approval and wallet signing
  6. SLEEP — Logs result, emits cycle_complete event, waits for next interval

Every step of every cycle is broadcast through the ThoughtStream — a singleton EventEmitter that pushes thoughts via WebSocket to the live dashboard in real time. Judges watching the demo see the agent's internal monologue as it happens.
